# 서울 데이터 AI(가칭) — 서비스 기획 v2

> 2026-04-09 업데이트 · 내부 피드백 반영

---

## 한 줄 요약

> 서울시 공공데이터를 AI 에이전트 팀이 미리 정제·구조화하고, 사용자의 질문이 들어오면 필요한 공개 데이터를 추가 탐색·조합·분석해 근거 있는 답변을 제공하는 서비스

---

## 문제

서울 열린데이터광장과 각종 공공 데이터 포털에는 방대한 데이터가 공개되어 있다.

하지만 일반 시민이 이를 실제로 활용하기는 매우 어렵다.

* 어떤 데이터가 있는지 찾기 어렵다
* 컬럼명이 영문 약어 중심이라 의미를 이해하기 어렵다
* 여러 데이터셋을 함께 쓰려면 조인, 정제, 시계열 정합성 검토가 필요하다
* 행정구역 단위, 기준 시점, 집계 방식이 달라 직접 비교하기 어렵다
* 결국 데이터는 공개되어 있어도 시민이 질문에 활용하기는 사실상 불가능하다

즉, **데이터는 열려 있지만 활용은 여전히 전문가 중심으로 닫혀 있다.**

---

## 해결책

이 서비스는 공공데이터를 단순히 검색하게 하는 것이 아니라,

**미리 정제된 데이터 카탈로그와 질문 시점의 에이전트 기반 데이터 탐색·분석을 결합해 시민이 이해할 수 있는 분석 결과로 바꿔주는 서비스**다.

핵심은 3단계 구조 + 에이전트 시뮬레이션 UX다.

---

## 1단계 — 데이터 사전 정제와 카탈로그 구축

먼저 서울시 공공데이터를 자동 스크립트와 LLM으로 분석해, 사람이 이해할 수 있는 형태의 데이터 카탈로그를 미리 구축한다.

### 정제 항목

* 컬럼명 한국어 매핑 (예: `CRADDR → 도로명주소`)
* 데이터셋 한 줄 요약
* 활용 가능한 질문 유형 정리
* 데이터 기준 단위 정리 (구 / 동 / 집계구 단위)
* 기준 시점과 갱신 주기 정리
* 다른 데이터셋과 연결 가능한 키 정의
* 품질 점수와 주의사항 표시
* 자주 쓰이는 검증된 데이터 조합 세트 관리

### 태깅 체계 (v2 추가)

카탈로그의 태그는 **이원 구조**로 관리한다.

| 구분 | 설명 | 예시 |
|------|------|------|
| **노출 태그** | 사용자에게 보이는 태그. 직관적이고 질문 연결 가능 | `육아`, `안전`, `교통`, `1인가구` |
| **내부 태그** | 시스템이 매칭에 사용하는 태그. 정밀하고 구조적 | `spatial:dong`, `join:SGG_CD`, `domain:childcare`, `quality:A` |

노출 태그는 사용자 검색·필터에 사용되고, 내부 태그는 에이전트가 데이터셋 선택·결합 시 사용한다.

### 초기 데이터셋 범위 (v2 추가)

전체 서울시 공공데이터를 다루는 것은 비현실적이므로, **초기에는 핵심 도메인 중심으로 선별**한다.

**초기 선별 도메인:**
* 육아 (어린이집, 유치원, 키즈카페)
* 안전 (CCTV, 가로등, 범죄 통계, 안전지수)
* 의료 (소아과, 응급의료, 약국)
* 교통 (지하철, 버스, 자전거)
* 생활 (편의점, 마트, 공원)
* 인구 (생활인구, 주민등록인구)

데이터셋 수는 **초기 15~20개** 수준으로 시작하고, 시나리오 확장에 따라 점진적으로 추가한다.

이 카탈로그는 단순 메타데이터가 아니라, **서비스의 핵심 자산이자 이후 탐색과 분석의 기반**이 된다.

---

## 2단계 — 질문 기반 데이터 탐색과 후보 조합 확장

사용자가 질문하면, AI는 먼저 사전 구축된 카탈로그에서 관련 데이터셋을 찾는다.

그다음 내부 카탈로그만으로 충분하지 않으면, 백엔드에서 추가 탐색을 수행해 질문에 더 적합한 공개 데이터를 찾는다.

### 예시

```text
시민: "마포구에서 아이 키우기 좋은 동네를 알려줘"
```

AI는 다음과 같이 동작한다.

1. **질문 의도 파악** — 지역: 마포구, 목적: 육아 친화 지역 비교
2. **내부 카탈로그 1차 탐색** — 어린이집, 공원, 소아과, 생활인구, 안전 지표
3. **백엔드 데이터 추가 탐색** — 보조 지표 필요 여부 판단, 관련 공개 데이터 후보 확장
4. **사용 가능성 검증** — 지역 단위, 시점, 결합 가능성, 품질 확인

이 단계는 단순 검색이 아니라, **질문에 맞는 데이터를 동적으로 발굴하고 분석 가능한 후보군으로 정리하는 과정**이다.

실제 구현에서는 속도와 안정성을 위해 무제한 탐색이 아니라, 신뢰 가능한 공개 데이터 소스와 정해진 탐색 범위를 중심으로 단계적으로 확장한다.

---

## 3단계 — AI 에이전트 기반 분석과 결과 제공

최종적으로 AI 에이전트 팀이 검증된 데이터셋 조합을 바탕으로 실제 분석을 수행하고 결과를 제공한다.

### 분석 범위 (v2 업데이트)

| 포함 | 미포함 (향후) |
|------|-------------|
| 데이터 정제·결합 | ML 모델 학습 |
| 통계 분석 (평균, 분포, 비율) | 딥러닝 예측 |
| **Feature Engineering** (파생 변수 생성, 정규화, 가중치 적용) | 실시간 스트리밍 분석 |
| 점수화·순위화 | GPU 기반 연산 |
| 시각화 생성 | |

> Feature Engineering까지 포함하되, ML 학습까지 가면 컴퓨팅 리소스 과부하가 우려되므로 초기에는 제외한다.

### 결과 화면 제공 정보

* 동별 비교 결과와 순위
* 지도 히트맵
* 사용한 데이터셋 출처
* 계산 기준과 가중치
* 기준 시점
* 해석 시 주의사항

LLM은 질문 해석, 데이터 탐색, 분석 흐름 설계 역할을 맡고, **실제 계산은 코드(Python)와 데이터 엔진이 수행**해 환각 가능성을 줄인다.

## 에이전트 시뮬레이션 UX (v2 신규)

### 개요

기존 "분석 결과를 준비하고 있습니다..." 같은 단순 로딩 대신, **데이터 분석 회사 시뮬레이션** 컨셉으로 대기 화면을 구성한다.

AI Town에서 영감을 받아, 각 분석 단계를 **에이전트 캐릭터가 출근·작업·핸드오프·퇴근하는 과정**으로 시각화한다.

### 에이전트 팀 구성

| 에이전트 | 역할 | 담당 단계 |
|---------|------|----------|
| 🔍 **Scout** | 데이터 탐색 에이전트 | 의도 파악 → 카탈로그 탐색 |
| 🧹 **Refiner** | 데이터 정제 에이전트 | 데이터 검증·결합 |
| ⚙️ **Forge** | 피처 엔지니어링 에이전트 | 파생 변수 생성·점수 계산 |
| 📊 **Canvas** | 시각화 에이전트 | 차트·지도·리포트 생성 |

### 흐름

```
Scout 출근 → 키워드 추출 → 데이터셋 검색 → Scout이 Refiner에게 전달 → Scout 퇴근
                                                ↓
Refiner 출근 → 컬럼 검증 → 결합 키 매칭 → Refiner가 Forge에게 전달 → Refiner 퇴근
                                                ↓
Forge 출근 → 파생 변수 생성 → 점수 계산 → Forge가 Canvas에게 전달 → Forge 퇴근
                                                ↓
Canvas 출근 → 차트 생성 → 지도 레이어 → 리포트 완성 → Canvas 퇴근 → 결과 표시
```

### UX 요소

* 각 에이전트에 아바타, 이름, 역할, 상태(대기/출근/작업/전달/퇴근) 표시
* 작업 중일 때 실시간 태스크 로그가 흘러감
* 에이전트 간 핸드오프 시 연결 애니메이션
* 하단에 전체 Activity Log
* **이 프레임은 다른 프로젝트에서도 재활용 가능한 범용 컴포넌트**로 설계

---

## 핵심 원칙

이 서비스는 "정답을 단정하는 추천 서비스"가 아니다.

대신 **설명 가능한 비교·탐색 서비스**를 지향한다.

모든 결과에는 아래 정보가 함께 제공된다.

* 어떤 데이터를 사용했는지
* 어떤 과정을 거쳐 데이터가 선택되었는지
* 어떤 기준으로 점수를 계산했는지
* 데이터 기준 시점이 언제인지
* 해석 시 어떤 한계가 있는지

---

## 핵심 차별점

| **기존 공공데이터 서비스** | **서울 데이터 AI(가칭)** |
| -- | -- |
| 데이터 검색·다운로드 중심 | 질문 기반 분석 결과 제공 |
| 단일 데이터셋 중심 | 여러 데이터셋의 동적 탐색·조합 |
| 영문 약어·복잡한 컬럼 구조 | 한국어 매핑과 의미 설명 제공 |
| 분석은 사용자 몫 | 실제 계산 결과와 시각화까지 제공 |
| 결과 근거 파악 어려움 | 데이터 출처, 선택 과정, 계산식 투명 공개 |
| 로딩 중 정보 없음 | **에이전트가 일하는 과정을 실시간으로 시각화** |

## 초기 서비스 범위

초기에는 모든 질문을 자유롭게 처리하기보다, **대표 시나리오 중심으로 시작**한다.

### 예시 시나리오

* 육아 친화 지역 비교
* 1인 가구 생활 편의 지역 비교
* 야간 귀가 안전도 비교
* 공원·의료·교통 접근성 비교

정확도와 신뢰성을 확보하면서, 향후 더 다양한 질문으로 확장한다.

## 기술 스택 (v2 신규)

| 레이어 | 기술 | 선택 이유 |
|--------|------|----------|
| **DB** | Supabase (PostgreSQL) | 무료 + PostgreSQL 기반 확장성 및 성능 안정성 |
| **데이터 분석 에이전트** | Python | 데이터 처리 및 Feature Engineering 생태계 최강 |
| **백엔드 API** | FastAPI | Python 기반으로 분석 에이전트와 동일 언어, 빠른 개발 |
| **프론트엔드** | Next.js + TypeScript + Tailwind + shadcn/ui | App Router, SSR, 컴포넌트 생태계 |
| **차트** | Recharts | React 네이티브, 반응형 |
| **지도** | React-Leaflet | 무료, GeoJSON 지원 |

### 아키텍처

```
[Next.js FE] → [FastAPI] → [Python Agent] → [Supabase (PostgreSQL)]
                  ↕                ↕
           WebSocket          서울시 공공데이터 API
        (에이전트 상태          (추가 데이터 탐색)
         실시간 전달)
```

## 화면 구성

### 1. 랜딩

* 검색창 중심
* 추천 질문 제시
* 서비스 핵심 메시지 강조

### 2. AI 분석 화면

* **좌측**: 대화, 사용 데이터, 분석 단계 인디케이터
* **우측 (분석 중)**: 에이전트 타운 시뮬레이션 — 에이전트가 출근·작업·핸드오프·퇴근하는 과정
* **우측 (완료 후)**: 지도, 차트, 비교 결과, 데이터 출처
* **하단**: 계산 기준, 가중치, 주의사항

### 3. 카탈로그

* 정제된 데이터셋 목록
* 노출 태그 기반 필터/검색
* 품질 점수, 갱신일, 연결 가능 데이터 표시

### 4. 데이터 상세

* 컬럼 한국어 매핑 테이블
* 활용 시나리오
* 결합 가능한 다른 데이터셋
* 단위, 시점, 품질 관련 안내

### 5. 지도 시각화

* 서울 구/동 드릴다운
* 지표별 레이어 토글
* 비교 결과와 원천 데이터 동시 확인

## 기대 효과

* 시민이 데이터 분석가가 아니어도 공공데이터를 활용할 수 있다
* 공개 데이터를 실제 질문과 의사결정에 연결할 수 있다
* 데이터 활용의 마지막 장벽을 낮출 수 있다
* 데이터 출처와 분석 과정을 투명하게 공개해 신뢰를 높일 수 있다
* **에이전트 UX 프레임은 다른 프로젝트에서도 범용적으로 재활용 가능**
